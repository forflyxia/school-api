ChannelManager对接过程

Skip to end of metadata
  • Created by vhh郝豪(Peter), last modified on Jan 23, 2017
Go to start of metadata

本文以RMS（GroupID=369）为例，记录ChannelManager对接过程中出现问题及相应的解决方法，以供其他CM对接和流程的完善提供参考。

测试环境对接阶段

 

前提：已在携程测试环境为该供应商创建或者关联一个CM测试酒店。

1.注意点

 

  • 一定要配置佣金！（2699）
  • 一定要关闭新订前可订检查的开关！（4257）
  • 一定要对接酒店的用户名和密码！（3223）
  • 一定要检查房型是否在黑名单，当查询接口没有吐出时！（4177）
  • 如果供应商没有单独推送最晚取消时间，则读取集团级别的配置

 

2. 步骤

2.1  创建用户名、密码

3223 模块创建Ebooking用户名、密码及集团代号

 

  • 用户和密码错误报错

 <Error Type="4" ShortText="invalid CodeContext." Code="175" />

  • 没有为对接酒店单独创建用户名和密码报错

 <Error Type="3" ShortText="This hotel is not active in Ctrip, please wait." Code="375" />

2.2 设置佣金

2699 模块进行佣金设置。

未设置佣金的表现为： 供应商推送给直连而直连缓存到DB中的房价房态和房量请求处理失败，具体表现为供应商看到推送到携程的房量成功，但房价房态没有更新到

参考下例链接进行酒店佣金设置：CM测试

 

建议：

a. 各个状态码表示什么含义？
b.一条message处理失败，能不能给些提示error message，一眼就看明白？能否不用再翻代码去debug出来？

2.3 集团配置

4257 模块

在配置时要注意，要把所有可订检查的开关全部关闭，否则会导致订单发送直连失败。

可参考生产环境的CM配置。

 

2.3  Web Service Address

测试环境 web service 地址：  http://vendor-ctrip.fws.ctripqa.com/Hotel/OTAReceive/OTAReceiveService.asmx

由于PDF文档（Ctrip Intergration API Specification for CM v2.5.0.pdf）未及时更新，其中的测试环境URL对方无法访问， 对接开始阶段告知对方正确的URL。

2.4 携程接口开发

2.5 测试预订接口

  •  现付订单

CM：应推送现付房价、房态和房量

携程：检查前端页面取消时间和取消政策，房价计算是否与预期一致。

 

  • 预付订单

CM：应推送预付房价、房态和房量

携程：检查前端页面取消时间和取消政策，房价计算是否与预期一致。

3. 问题及解决

3.1  用户名和密码问题

如2.1 中描述。

3.2 订单无法发送直连

现象：online页面下单成功，但是通过后台4257模块发现此订单无法发送直连，挖掘DB log 其中错误信息描述出现 “NOROOM,酒店满房” 字样。
分析：　

a. 从poly的代码逻辑中，发现在预订过程中的service调用堆栈并且有RQ和RS的开关控制，所以可以在后台打开log（IsCallSoaRecordLog）开关帮助追踪问题在哪里。



IsCallSoaRecordLog 默认为F，打开后可以记录订CM的房间时，通过log,可以观察到预订过程在PolyService中的服务调用堆栈。
b.  从log中可以抓取到一次页面预订先后调用了 OrderPostSendProcess, SubmitHotelOrderPoly, SendNewOrder的请求。


SendNewOrder RS:

<Response>
<Header ShouldRecordPerformanceTime="false" ResultCode="Success" RequestBodySize="0" SerializeMode="Xml" RouteStep="0" />
<SendNewOrderResponse>
<SendStatus>3</SendStatus>
<IsOrderSendType2>F</IsOrderSendType2>
<IsRemarkSendType2>F</IsRemarkSendType2>
<IsOrderStatusUpdated>T</IsOrderStatusUpdated>
<ReturnCode>1</ReturnCode>
<ReturnDescript>NOROOM,酒店满房</ReturnDescript>
<InterFaceConfirmNO />
</SendNewOrderResponse>
</Response>



从SendNewOrder请求中，抓取请求发在本地进行调试InterfaceSwitchPolyService，发现在CM预订过程中竟然调用了可订检查接口！而CM没有实现可订检查也不需要实现的。至此发现问题出在4257的配置上面： 订单发送前是否检查可订，应该设置成“否”。



总结： 在配置CM时，一定要确保配置正确，还要注意扩展配置。可参考生产环境中的配置。
注意事项：

1)  CM没有可订检查，这点我们会注意到，但订单前的可订检查也要注意开关关态。

2)  扩展配置项中的log开关很有用，可帮助定位问题在哪里。

 

3.3 接口查询不到预期的子房型

现象： 测试酒店有很多子房型，但是调用OTA_HotelRatePlanRQ接口，返回的RatePlanCode 列表怎么只有两个呢？

原因：接口吐出房型被加入黑名单或者暴露权限未打开。

分析：
原来OTA_HotelRatePlanRQ 只允许AllowViewRate = 1 的房型露出来。

通过下列sql查看子房型的暴露程度权限：  SELECT *
FROM hotelinterfacedb.dbo.HotelInterface_PermissionRoom(NOLOCK) 
WHERE Hotel = 432126;


AllowUpdateAvailablity : 允许更新房态
AllowUpdatePrice: 允许更新房价
AllowUpdateInventory: 允许更新房量
AllowViewRate:允许启用新Rate ?表示什么？

通过此表（子房型暴露权限表），定义了子房型黑名单，具备两种类型的属性值中被认为列入了黑名单：

第一种：房价、房态和房量都不允许更新，即可认为此子房型被列入了黑名单。

第二种：

(1) 此酒店启用了NewRate,即在HotelInterface_PermissionRoom表中只要存在一个房型的AllowViewRate为1 即表示启用NewRate.

(2) 有一个房型的AllowViewRate属性值为0。

在符合条件（1）和（2），即可认为此子房型（AllowViewRate = 0 && 启用了NewRate）被列入了黑名单。

4177模块可以设置黑名单：http://10.2.57.67/interfaceplatformmanage/VenderStaticData/SelectHotelInterfacePermissionHotel.aspx
点击“设置房型黑名单” 或者“设置Rate关联”。

    1. 没有在黑名单的房型，HtlRoomsPubShardDB中有房价房态和房量，可是在online页面展示出来，这是为什么？

３.4  CM订单在5780取消失败

Interface Adapter收到取消请求时，如果请求中传递了GroupID并且GroupID> 0，依赖4257模块管理的配置表来判断该订单否是CM订单；如果GroupID=0，则从DB中直接读取GroupID是否属于CM。

若是CM则走CM Adapter ， 否则，找不到对应Adapter而无法发送取消订单，从而引起5780取消失败报错。

3.5 房价房态更新不成功

具体表现为供应商看到推送到携程的房量成功，但直连缓存后再提取处理的message处理失败，所以房价房态没有更新到大系统。

原因分析：经过debug发现，没有为该酒店设置佣金，需要在2699模块进行佣金设置，可参考其他CM配置。

参考下例链接进行酒店佣金设置：CM测试

4. 对接建议

  • 前期准备：联调开发阶段的准备工作是在测试环境中进行，携程应标准化对接流程，除API文档发送给供应商外，在对接初始段应书面告诉对方联调用 的webservice地址。
  • 标准流程： 应有专门文档指示对接注意事项，先为供应商创好测试酒店及对应的Ebooking用户名和密码。 这些应该通过文档的形式固定下来，而不是一旦安排人员负责对接时，一遍又一遍地踩坑。 
  • 错误处理：完善错误处理机制，特别地，增加Error Code 及Error Message, 通过友好地方式反馈给供应商。message为什么处理失败？什么原因导致的？我们在联调用过程中， 花了很多时间和精力来解答这类问题，常常需要翻代码debug.

做到这些，相应能快速给供应商进行反馈，加快对接进度。

生产环境 Pilot

  •  CM供应商，提供endpoint URL ,用户名和密码

  • 负责后台模块配置（可联系 @春华）
  • 携程业务人员，提供Ebooking用户名和密码，及相应集团代号，这些信息需要提供给CM供应商。
  • 技术开发检查配置，并调用OTA_HotelRatePlan接口以检查是否能获取到子房型信息。

    生产环境服务地址：https://vendor.ctrip.com/Hotel/OTAReceive/OTAReceiveService.asmx
